# push前のシークレット・内部ID確認

publicリポジトリへのpushは、マージ前のPR diffも含めて全世界から見える。一度pushすると、GitHub側やクローラーに取得された可能性を消せないため、push前に止めることに意味がある。

これは個人の作り方であり、会社としての規定ではない。実行前に、対象リポジトリの `CLAUDE.md` に固有ルールがあればそれを優先する。

## (1) push先の公開範囲を確認

- `pwd` でCWDが対象リポジトリのトップであることを確認する。
- `gh repo view --json visibility -q .visibility` で対象リポジトリのvisibilityを確認する。
- `private`/`internal` なら (2)(3) の詳細スキャンは省略してよい。`public` なら次に進む。

## (2) 今回pushする範囲を特定する

- 追跡branchがある場合: `git rev-list @{u}..HEAD` で対象commitを列挙し、`git diff @{u}..HEAD` で差分を見る。
- 新規branchで追跡先がない場合（初回push/PR作成前）: default branch名を決め打ちせず確認した上で、`git diff origin/<default>...HEAD`（マージベース差分、PRのdiffと一致する）を見る。
- 対象範囲全体を見る。直近1commitだけでなく、今回pushするcommit全部が対象。

## (3) 2種類の観点でスキャンする

自動スキャンだけでは不十分。汎用ツールは「鍵っぽい形式の文字列」は拾えるが、「UUID型の内部IDが実は非公開だった」ような文脈依存の漏えいは拾えない。両方を行う。

- **自動スキャン（既知の認証情報形式）**: `gitleaks` が使えるなら `gitleaks git --log-opts="<対象range>"` で対象commit範囲を直接スキャンする。未インストールなら `mise x gitleaks@latest -- gitleaks git --log-opts="<対象range>"` でグローバル設定を汚さず一時実行する。AWSキー・秘密鍵・GitHub/Slackトークンなど既知パターンを検出する。
- **手動レビュー（内部ID・文脈依存の情報）**: (2) の diff を自分で読み、次のような「ツールでは拾えない」情報がないか確認する。
  - クラウドのアカウントID・zone ID・tunnel ID・project ID・ARNなどのリソース識別子
  - 内部ホスト名・IPアドレス・非公開のURL
  - パスワード・APIキー（高エントロピーでない簡易なものも含む）
  - 個人名・メールアドレスなど、公開を意図していない情報
  - 疑わしい値を見つけたら、それが**別の経路で既に公開済みか**を確認する（例: DNSレコード・既存の公開ドキュメント・`curl`/`dig` で実際に引けるか）。既に公開済みならリスクは低いが、非公開だった場合は今回のpushで初めて公開されることになる。

## (4) 判断

- 認証情報（トークン・パスワード・鍵）が見つかった場合: pushを止め、依頼者に確認する。redactして再push、または該当の認証情報自体をローテーションする必要がないか検討する。
- 内部IDなど文脈依存の情報が見つかった場合: (3) の確認結果を添えて依頼者に報告し、指示を仰ぐ。実害が小さくても、公開するかどうかは依頼者の判断に委ねる（黙って公開しない・黙って書き換えない）。
- 何も見つからなければ、通常通りpushを実行する。

## (5) 既にpushしてしまっていた場合

- 対象branchがPRとしてまだ未マージなら、該当commitをamendまたは新しいcommitで修正し、`git push --force-with-lease` で該当branchのみ上書きする。`main`/`master`への force pushは別途強い確認を要する。
- 既にマージ済みで default branch に取り込まれている場合、履歴の書き換えは影響が大きいため、まず依頼者に相談する。

## よくある取りこぼし

| 取りこぼし | 正しくは |
|---|---|
| privateだからと油断して他リポジトリでも省略する | (1) でvisibilityを毎回確認してから判断する |
| 直近1commitのdiffだけ見る | 今回pushする範囲全部（`@{u}..HEAD` など）を見る |
| 自動スキャンツールの結果だけで安心する | 内部ID・文脈依存の情報は手動レビューでしか拾えない |
| UUID的な値を「意味のある文字列じゃないから安全」と判断する | 別経路で既に公開済みかを確認してから判断する |
| 疑わしい値を見つけて黙って書き換える／黙って公開する | 依頼者に確認してから対応する |
| force pushで済ませて終わりにする | 既にpush済みの内容が外部に取得された可能性は消せないと認識する |
